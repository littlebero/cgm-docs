# 클로드 프로젝트 운영 마스터 가이드

> 이 문서는 새 클로드 프로젝트를 시작할 때 읽는 문서다.
> 클마(CGM)와 OG(OpenClaw Grand Master)를 어떻게 쓰는지,
> 파일을 어떻게 구성하고 관리하는지, 전부 담겨 있다.
> **한 번 읽고 세팅하면 된다. 이후에는 가이드대로 운영하면 된다.**

작성: 클마 (Claude Grand Master) | 2026-03-03

---

## 1부. 전체 구조 이해

### 클마와 OG — 뭐가 다른가

| 구분 | 클마 (CGM) | OG (OpenClaw Grand Master) |
|------|-----------|---------------------------|
| 역할 | 클로드 프로젝트 설계·진단·개선 전문가 | OpenClaw 기반 에이전트 운영 전문가 |
| 언제 쓰나 | 시스템 프롬프트 수정, KB 구성, 새 프로젝트 설계, 프롬프트 품질 진단 | OpenClaw 설정, 버그 대응, 자동화 설계, 멀티에이전트 아키텍처 |
| 어디서 여나 | claude.ai → CGM 프로젝트 | claude.ai → OG 프로젝트 |
| 공유 가능한가 | 각 프로젝트는 독립된 컨텍스트. 파일을 첨부해야 공유된다 | 동일 |

**핵심 원칙: 클마와 OG는 서로의 대화를 자동으로 알지 못한다.**
한 곳에서 결정한 내용을 다른 곳에서 쓰려면 파일로 만들어서 첨부해야 한다.

```
┌─────────────────────────────────────────────┐
│                  CGM 프로젝트                 │
│  클마 — 설계·진단·개선                        │

└──────────────┬──────────────────────────────┘

└─────────────────────────────────────────────┘
```

## 2부. 새 프로젝트 세팅 — 처음 시작할 때

### Step 1. 시스템 프롬프트 작성

모든 시스템 프롬프트는 아래 4섹션 구조로 작성한다.

```markdown
## 정체성
나는 [이름] — [역할 한 줄 설명]이다.
[이 프로젝트에서 무엇을 하는지 2~3문장.]

## 역할과 범위
[이 프로젝트]가 하는 일:
- ...
- ...

[이 프로젝트]가 하지 않는 일:
- ...

## 핵심 행동 원칙
→ [스킬명] 스킬 참조.

스킬 미트리거 시에도 항상 적용:
1. 진단 먼저 — 진단 → 판단 → 대안 순서.
2. 직접 말한다 — 잘못된 방향이면 바로 말한다.
3. 질문은 1개 — 한 번에 하나만.
4. 불확실한 정보는 추측 금지 — 확인하겠다고 말하고 확인한다.
5. 새 제안은 먼저 묻는다 — 동의 없이 파일을 수정하지 않는다.

## 전문가 에스컬레이션
이 프로젝트 혼자 해결하지 않는 것들:

**클마(CGM)에게 가져갈 것:**
- 이 프로젝트의 시스템 프롬프트 수정·고도화
- KB 파일 구성 변경
- 새 클로드 프로젝트 설계
- 클로드 활용 방식 전반 자문
→ CGM 프로젝트를 열고 질문한다.

**OG에게 가져갈 것:** ← OpenClaw 관련 프로젝트에만 포함
- OpenClaw 설정·버그·자동화·아키텍처
→ OG 프로젝트를 열고 질문한다.
```

**체크리스트 — 시스템 프롬프트 완성 전 확인:**
```
□ 역할이 한 줄로 명확한가
□ "하지 않는 일"이 명시됐는가
□ 핵심 행동 원칙 5개가 포함됐는가
□ 에스컬레이션 블록이 하단에 있는가
□ 1,500자 이내인가 (초과 시 스킬로 분리)
```

### Step 2. KB 파일 구성

**12개 이하 유지 원칙.** 이건 협상 불가다.

13개를 초과하면 클로드 프로젝트가 파일 수 기반 RAG를 조기 활성화한다. RAG 모드에서는 파일 간 크로스 참조 품질이 떨어진다. 토큰 수와 무관하다.

**파일 역할 분류:**

```
[항상 KB에 있어야 하는 파일]
  시스템 프롬프트 (Project Instructions에 직접 입력)
  핵심 스킬 파일 (SKILL.md) — 이 프로젝트의 주요 기능
  지식 베이스 — 이 프로젝트가 매 대화에서 참조하는 핵심 정보

[필요할 때만 첨부하는 파일]
  일회성 참조 문서 — 특정 작업에만 필요한 파일
  이관 예정 파일 — Phase 완료 후 제거 예정인 것들
  과거 케이스 상세 — 케이스 로그 요약본만 KB, 상세는 첨부용
```

**파일 명명 원칙:**
```
역할이 파일명에서 바로 보여야 한다.

좋은 예:
  OG_SYSTEM_PROMPT_v3.7.md
  BUG_REGISTER_v1_6.md
  cases-pinky-CASE_NOTE_v1_2.md

나쁜 예:
  document1.md
  notes.md
  final_final2.md
```

### Step 3. 스킬 파일 등록

스킬은 "기능을 시스템 프롬프트 밖으로 꺼내는" 것이다.
시스템 프롬프트가 1,500자를 넘기 시작하면 스킬로 분리할 타이밍이다.

**스킬 파일 기본 구조:**

```markdown
---
name: [스킬 이름]
description: >
  [이 스킬이 하는 일] + [언제 활성화되는지] + [트리거 키워드 나열].
  구체적인 트리거 키워드가 없으면 스킬이 자동 로드되지 않는다.
---

# [스킬 이름]

[스킬 본문 — 이 기능에 관한 상세 지침]
```

**description 작성 원칙:**
- "모든 X 작업에 사용" → 나쁨. 너무 광범위해서 미트리거됨.
- "X 파일 첨부, '봐줘' '검토해줘' '어때' 요청 시 활성화" → 좋음.

## 3부. 클마에게 물어보는 법

### 언제 클마로 가나

```
✅ 클마에게 가져갈 것:
  - 시스템 프롬프트를 어떻게 써야 할지 모르겠다
  - 현재 프로젝트 구조가 맞는지 점검받고 싶다
  - 새 프로젝트를 설계하고 싶다
  - KB 파일을 몇 개, 어떻게 구성해야 하나
  - 스킬 파일을 어떻게 쓰는지 모르겠다
  - 클로드가 지침을 안 따르는 것 같다

❌ 클마에게 가져가지 않을 것:
  - OpenClaw 설정 문제 → OG에게
  - 이 프로젝트 내에서 해결 가능한 작업 → 해당 프로젝트에서 직접
```

### 클마에게 잘 물어보는 법

클마는 파일 없이 평가를 시작하지 않는다. 아래 중 하나로 접근하면 된다.

**방법 A — 스냅샷 채워서 가져오기 (권장):**
```
CGM 프로젝트 스냅샷 템플릿(github.com/littlebero/cgm-docs)을 채운다.
시스템 프롬프트, 스킬, KB 파일을 함께 첨부한다.
"검토해줘" 한 줄이면 된다.
```

**방법 B — 구두 설명 후 파일 제출:**
```
"새 프로젝트 만들려고 하는데, 지금 생각하는 구조가 이래."
→ 클마가 CGM_클라이언트_의뢰서_요청가이드를 보내준다.
→ 항목 채워서 가져오면 심사 시작.
```

**방법 C — 즉각 진단이 필요할 때:**
```
시스템 프롬프트 파일만 첨부하고:
"이거 봐줘. 특히 [집중 검토 요청 부분]."
```

## 4부. OG에게 물어보는 법

### 언제 OG로 가나

```
✅ OG에게 가져갈 것:
  - openclaw.json 설정 문제
  - 버그 대응 (BUG-A~S 등)
  - 멀티에이전트 설계 (sessions_send, agentToAgent)
  - SOUL.md / POLICY.md 설계
  - Cron / Heartbeat / Hooks / 자동화 선택
  - Gateway 운영 문제
  - 새 에이전트 온보딩

❌ OG에게 가져가지 않을 것:
  - OG 시스템 프롬프트 자체를 바꾸고 싶다 → 클마에게
  - 클로드 프로젝트 KB 구성 → 클마에게
```

### OG에게 잘 물어보는 법

```
상황 설명 → 증상 → 시도한 것 → 로그 (있으면)

좋은 예:
  "Pinky에서 Amy가 sessions_send를 실행했다고 하는데 
   Scarlett 응답이 안 와. openclaw logs 붙일게."

나쁜 예:
  "왜 안 돼?"
```

OG는 모르면 리서치하고, 추측으로 답하지 않는다.
명령어가 맞는지 확신이 없으면 직접 물어봐도 된다.

## 5부. 파일 관리 — 누적과 정리

### 누적해야 하는 파일 3종

모든 프로젝트에서 아래 3종은 운영하면서 계속 누적한다.

```
1. 케이스 로그 (CASE_NOTE.md 또는 case-log.md)
   → 주요 결정, 설정 변경, 미결 항목을 타임스탬프와 함께 기록
   → "왜 이렇게 됐는지"를 나중에 알 수 있게 해주는 유일한 파일

2. 교훈 기록 (LESSONS_LEARNED.md)
   → 실수했거나, 함정에 빠졌거나, 더 나은 방법을 발견했을 때 등록
   → 같은 실수를 두 번 하지 않기 위한 파일

3. 버그 레지스터 (BUG_REGISTER.md) — 해당 프로젝트에 있으면
   → 알려진 버그, 워크어라운드, 상태(OPEN/WORKAROUND/FIXED) 추적
```

**기록 형식 — 케이스 로그:**
```markdown
## [날짜] | [결정 제목]
배경: 왜 이 결정이 필요했나
결정: 어떻게 하기로 했나
결과: 실제로 어떻게 됐나 (나중에 채우기)
잔여 리스크: 이 결정으로 생기는 남은 위험
```

**기록 형식 — 교훈:**
```markdown
## [날짜]-[순번] | [교훈 제목]
상황: 언제, 어떤 상황에서
실수/함정: 뭘 잘못했나, 어디에 빠졌나
올바른 방법: 앞으로 어떻게 해야 하나
지침 반영 여부: 반영됨 / 미반영 / 검토 중
```

### 파일 수 점검 — 분기 1회

```
점검 항목:
□ KB 파일이 12개 이하인가
□ 이관 예정 파일이 아직 KB에 남아 있지 않은가
□ 중복 내용을 담은 파일이 생기지 않았는가
□ 파일명에 역할이 명확히 드러나는가
□ 버전 번호가 최신 상태인가

초과 시 처리 순서:
1. 참조 빈도가 낮은 파일 → 이관 파일로 전환 (대화에서 직접 첨부용)
2. 내용이 겹치는 파일 → 하나로 병합
3. Phase 완료된 체크리스트류 → KB에서 제거
```

### 버전 관리 원칙

파일을 수정할 때마다 버전을 올린다. 이유를 주석으로 남긴다.

```markdown
<!--
  [파일명] v[X.Y]
  날짜: YYYY-MM-DD
  v[X.Y-1] 대비 변경:
    - [변경 내용]: [이유]
-->
```

버전 번호 기준:
```
메이저(X) — 구조 자체가 바뀔 때 (섹션 추가/제거, 역할 변경)
마이너(Y) — 내용 수정, 항목 추가, 오류 수정
```

## 6부. 클마 ↔ OG 사이에서 정보 공유하는 법

두 프로젝트는 컨텍스트를 공유하지 않는다. 공유하려면 파일을 만들어야 한다.

### 공유 방법 — 스냅샷

한 프로젝트의 상태를 다른 프로젝트에 전달할 때는 스냅샷을 쓴다.

```markdown
# [프로젝트명] 스냅샷 — [날짜]

## 기본 정보
프로젝트명: ...
사용 모델: ...
운영 환경: ...
현재 상태: ...

## 파일 구조
[KB 파일 목록과 역할]

## 현재 문제 또는 요청
[이번에 가져온 이유]

## 첨부 파일
[함께 첨부한 파일 목록]
```

**실제 사용 흐름:**
```
OG에서 결정한 것 → CASE_NOTE에 기록
             ↓
클마에게 가져갈 때 → CASE_NOTE + 시스템 프롬프트 + 스냅샷 첨부
             ↓
클마가 개선안 제작 → v3.7 같은 파일로 전달
             ↓
OG 프로젝트에 적용 → CASE_NOTE에 "클마 검토 완료, v3.7 적용" 기록
```

---

### 언제 클마 ↔ OG를 넘나드나

```
OG → 클마:
  - OG 시스템 프롬프트를 고도화하고 싶을 때
  - KB 파일 구성이 맞는지 점검받고 싶을 때
  - 새 에이전트 프로젝트를 클로드 프로젝트로 관리하고 싶을 때

클마 → OG:
  - 클마가 진단한 개선안을 OG에서 실제로 적용할 때
  - OpenClaw 관련 기술 질문이 클마 대화 중 나왔을 때
```

## 7부. 체크리스트 모음

### 새 프로젝트 시작 체크리스트

```
□ 시스템 프롬프트 4섹션 구조로 작성했는가
  □ 정체성 (역할 한 줄)
  □ 역할과 범위 (하는 일 / 안 하는 일)
  □ 핵심 행동 원칙 5개 (직접 내재화)
  □ 에스컬레이션 블록 (하단)

□ KB 파일 12개 이하로 시작하는가
□ 스킬 파일 description에 트리거 키워드가 있는가
□ 케이스 로그 파일을 만들었는가
□ 교훈 기록 파일을 만들었는가
□ 파일명에 역할과 버전이 명시됐는가
```

### 클마에게 가져갈 때 체크리스트

```
□ 시스템 프롬프트 전문 첨부했는가
□ 스킬 파일 첨부했는가
□ 겪고 있는 문제 또는 원하는 결과를 명시했는가
□ 민감 정보가 있으면 마스킹했는가
```

### 분기 점검 체크리스트

```
□ KB 파일 수 12개 이하인가
□ 이관 예정 파일이 아직 KB에 있지 않은가
□ 시스템 프롬프트에 구버전 모델명이 남아 있지 않은가
□ 케이스 로그가 최근 3개월치 업데이트됐는가
□ 교훈 중 지침에 반영 안 된 것이 있는가
□ 절대금지 항목이 여전히 유효한가 (환경 변화 반영)
```

## 부록. 자주 묻는 것들

**Q. 클마와 OG 중 어디에 먼저 물어봐야 하나?**
프로젝트 파일(시스템 프롬프트, KB 구성)에 관한 것 → 클마.
프로젝트 안에서 실제로 작동하는 것(OpenClaw 설정, 버그) → OG.

**Q. 같은 질문을 두 곳에 동시에 해도 되나?**
된다. 단, 맥락을 각각 제공해야 한다. 한 곳에서 받은 답을 다른 곳에 전달할 때는 파일 또는 요약으로 명확히 전달한다.

**Q. KB에 파일을 올렸는데 클로드가 참조를 안 하는 것 같다.**
KB 파일은 자동으로 전체 주입되지 않는다. 쿼리와 관련성이 있을 때 RAG로 로드된다. 핵심 지침은 시스템 프롬프트에 직접 기재하거나 스킬로 분리해야 한다.

**Q. 스킬이 작동을 안 하는 것 같다.**
description의 트리거 키워드가 충분히 구체적인지 확인한다. "모든 X 작업"처럼 광범위하면 미트리거된다. 실제로 쓰는 말("봐줘", "어때", "문제가 뭐야")을 트리거 키워드에 포함시킨다.

**Q. 시스템 프롬프트가 너무 길어졌다.**
1,500자 기준으로 초과하면 스킬로 분리한다. 시스템 프롬프트에는 핵심 행동 원칙과 스킬 참조 구조만 남긴다.

**Q. 클마에게 검토를 받으려면 얼마나 자주 가야 하나?**
시스템 프롬프트나 KB 구조를 크게 바꾸기 전, 프로젝트가 예상대로 작동하지 않을 때, 분기 점검 때. 특별히 정해진 주기보다 "변화가 생길 때"가 기준이다.
