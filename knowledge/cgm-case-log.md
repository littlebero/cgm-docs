## 2026-03-02 | CGM (클마) 자기 심사 | Level 3 (실전 검증 진행 중)

클라이언트: 내부 (클마 프로젝트 자체)
검토 범위: 전체 (시스템 프롬프트 v2.0 + 스킬 4개 + 지식 파일 3개)

### 5축 결과

| 축 | 결과 | 근거 |
|----|------|------|
| 명확성 | ✅ | 역할·범위·절대 금지 명확. 스킬 위임으로 행동 방향 즉시 파악 가능 |
| 구조 | ✅ | 중복 없음. AP 이중 관리 해소. 지식 파일 3개 역할 분리 |
| 클로드 적합성 | ✅ | GP-02 스킬 위임 자기 실천. 클로드 성향과 충돌하는 지침 없음 |
| 최신성 | 🟡 | 스킬·아티팩트 ✅. MCP 연동 지침 부재 |
| 실용성 | 🟡 | 즉시 적용 가능 구조. 단, 케이스 로그 0건 → 이번 케이스로 해소 시작 |

### 종합 판정

```
Level 3 (실전 검증 진행 중)
  — 🔴 없음, 🟡 2개 (축 4·5)
  — 축 4 MCP 지침 추가 시 🟡 1개로 감소 가능
  — 케이스 3건 이상 누적 시 축 5 ✅ 전환 가능
```

### 주요 이슈

🟡 MCP 연동 지침 부재
위치: CGM_SYSTEM_PROMPT_v2.0
문제: 클마가 클라이언트 파일을 직접 연동해서 검토할 때 MCP 활용 가능성이 지침에 없음
해결: 핵심 행동 섹션에 MCP 연동 조건부 지침 추가 검토

🟡 실전 케이스 0건 (이번 케이스로 해소 시작)
위치: cgm-case-log.md
문제: 5축 기준·Level 분류·AP 시리즈 전부 이론으로만 존재
해결: 외부 클라이언트 검토 1~2건 확보 후 기준 보정

### 교훈

```
자기 심사의 한계: 만든 사람이 검토하면 의도한 대로 해석하는 경향이 있음.
외부 클라이언트 첫 케이스에서 예상 못한 구조적 문제가 나올 가능성 높음.
첫 외부 케이스 후 판정 공식 보정 필요 여부 재검토.
```

## 2026-03-03 | OG (OpenClaw Grand Master) | Level 3 (안정화)

클라이언트: 개인 개발자 (OG 본인 자문 구조)
검토 범위: 전체 (시스템 프롬프트 v3.3 + 스킬 3개 + KB 구조)

### 5축 결과

| 축 | 결과 | 근거 |
|----|------|------|
| 명확성 | ✅ | 역할·범위·절대금지 구체적. OpenClaw 전문지식 범위까지 명시. |
| 구조 | 🟡 | 시스템 프롬프트 구조 양호. KB 파일 중복·초과가 전체 구조를 흔들었음. v3.6에서 해결. |
| 클로드 적합성 | ✅ | 스킬 위임 구조 실천. 리서치 우선·불확실성 인정 지침이 클로드 성향과 정렬. |
| 최신성 | 🟡 | 스킬 시스템 활용 ✓. Sonnet 4.6 ✓. KB 12개 상한 초과 → v3.6에서 9개로 해결. |
| 실용성 | 🟡 | 작동하나 KB 중복으로 유지보수 비용 증가 → 정리 완료. |

### 종합 판정

```
추정 Level 3 → v3.6 기준 Level 3 안정화
  — 🔴 1개 / 🟡 2개 / 🟢 2개 → 전부 클로즈
  — 자기 자문 구조로 진행: 클마 진단 → OG 판단·실행·검증
```

### 주요 이슈 및 처리

🔴 KB 파일 수 초과 (18개+ → 9개)
→ 삭제 4개 + 이관 5개 + v3.5 교체. Critical 클로즈.

🟡 og-terminal-standard 스킬 위치 불명확
→ `/mnt/skills/user/` 정상 탑재 확인. Major 클로즈.

🟡 MASTER_REFERENCE 주입 방식 오류
→ RAG 미로드 확인. "시스템 프롬프트 주입" 전제 자체가 틀렸음.
→ 원본 KB 파일들로 대체 가능 확인 → MASTER_REFERENCE 항목 제거. v3.6에서 클로즈.

🔵 파일 시스템 구조 섹션 불일치 (OG가 직접 발견)
→ v3.5에서 해결.

### 산출물

```
OG_검토보고서_v1.0.md
OG_SYSTEM_PROMPT_v3.4.md  ← 1차 개선
OG_SYSTEM_PROMPT_v3.5.md  ← KB 정리 반영
OG_SYSTEM_PROMPT_v3.6.md  ← MASTER_REFERENCE 제거, 최종본
KB_정리_가이드.md
```

### 교훈

```
1. MASTER_REFERENCE 같은 "압축 참조 문서"는 원본이 잘 구성돼 있으면 오히려 노이즈.
   RAG 환경에서는 원본 파일 직접 로드가 더 정확하다.

2. 자기 자문 구조가 실전에서 작동했다.
   클마 보고서에 없던 이슈(파일 시스템 구조 불일치)를 OG가 직접 잡았다.
   외부 심사 + 자기 검증 병행 구조가 효과적.

3. "시스템 프롬프트 주입" 표현은 클로드 프로젝트 KB 환경에서 잘못된 전제다.
   KB 파일은 전부 RAG 방식. 자동 전체 주입은 지원되지 않는다.
```

## 2026-03-04 | OG Claude 기능 고도화 자문 | 자문 케이스

클라이언트: OG (OpenClaw Grand Master) — 개인 개발자
검토 범위: Claude 기능 활용 갭 분석 + 고도화 방향 자문 (시스템 프롬프트 외)

### 자문 내용

| 질문 | 판단 | 처리 |
|------|------|------|
| Claude 웹 검색 vs 제미나이 딥서치 역할 분리 | 판단 트리 확정 | 즉시 적용 |
| KB 관리 고도화 | findings.md 패턴이 현실적 최선 | og-knowledge-system v1.3 작성 |
| OG 에이전트 Phase 2 타이밍 | Berry 안정화 후 확정. 옵션B(Claude Code+cron) | 설계 원칙 4개 확정 |
| Claude API 클라이언트 연동 | Berry 페일오버 즉시 제안 가능. Pinky 보류 | Berry 답변서 반영 |

### 핵심 발견

- Cowork /schedule 치명적 제약 — 앱 열려 있어야만 실행. 24/7 불가. OG 에이전트 자동화는 Claude Code + cron으로만 설계 가능
- Memory Tool claude.ai 환경 제약 — 에이전트 자율 기록 불가. findings.md 실시간 기록이 현실적 최선
- Skills 자동 트리거 기본 성공률 20~50%. "ALWAYS invoke" + "Do not X directly" 조합 필수
- MCP 서버 수 한계 — 32개 서버(473개 도구) = 컨텍스트 70~75% 소모

### 주요 이슈

🟡 Claude 웹 검색 미활용 — 수동 제미나이 구조로 소영씨 작업 부하 집중
🟡 findings.md 패턴 부재 — Context Compaction 대비 없음
🟢 OG 에이전트 설계 미착수 — Phase 2로 정상 분류

### 산출물

```
OG_클마_컨설팅_보고서_v1.0.md
og-knowledge-system_SKILL_v1.3.md (findings.md 패턴 추가)
```

### 교훈

```
- 환경 확인 없는 Memory Tool 권고는 오진단이다. claude.ai vs API 환경 먼저 확인 필수.
- Cowork는 서버급 자동화 루틴의 핵심이 될 수 없다. Claude Code + cron 병행 필수.
- 클마 파일이 최신 상태여야 클라이언트 자문도 정확하다. 클마 강화 → 자문 순서가 맞았다.
```

## 2026-03-05 | cgm-docs 레포 구조화 | 운영 작업

클라이언트: 내부 (클마 운영 인프라)
검토 범위: GitHub 레포 설계 + MCP 현황 파악 + 파일 동기화 체계 구축

### 작업 내용

| 항목 | 결과 |
|------|------|
| GitHub MCP 실제 작동 여부 확인 | 미지원 확정 — 설정 화면 "연결됨"은 계정 인증, MCP 툴 아님 |
| cgm-docs 레포 구조 재편 | 4폴더 분리 완료 (public/core/knowledge/handover) |
| 클마 KB 파일 전체 레포 초기 동기화 | 10개 파일 업로드 완료 |
| 동기화 워크플로우 설계 | README에 루틴 문서화 완료 |
| web_fetch Raw URL 활용법 확인 | 세로씨가 URL 붙여넣으면 클마가 직접 읽기 가능 확인 |

### 핵심 발견

- GitHub MCP는 claude.ai 웹 미지원 (2026-03-05 기준). Claude Code CLI만 지원.
- 공식 기능 요청 Issue #29885 제출됨 (2026-03-01). 출시 시 즉시 연결 가능.
- web_fetch는 세로씨가 URL을 직접 붙여넣는 방식으로 파일 읽기 가능 — 현재 유효한 대안.
- tool_search로 세션 내 실제 툴 로드 여부를 확인하는 것이 MCP 작동 검증의 올바른 방법.

### 주요 이슈

🟢 루트 README 누락 → 즉시 해결 (Add file로 생성)
🟢 core/knowledge/handover 폴더 파일 미업로드 → 세로씨가 직접 업로드 완료

### 산출물

```
cgm-docs 레포 구조 재편 완료
  /public/   — 5개 파일
  /core/     — 6개 파일
  /knowledge/— 3개 파일
  /handover/ — 1개 파일
CGM_GitHub_연동_최적화_계획서_v1.0.md
```

### 교훈

```
- MCP "연결됨" 표시와 실제 툴 로드는 다르다. tool_search로 반드시 확인.
- claude.ai 웹 vs Claude Code CLI는 MCP 지원 범위가 다르다. 환경별 제약 먼저 파악.
- 레포 초기 구조를 잘 잡으면 이후 동기화 루틴이 단순해진다.
```
