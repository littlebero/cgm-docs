# CGM Prompt Patterns

클마가 검토하며 발견한 좋은/나쁜 프롬프트 패턴 모음.
패턴 발견 시마다 누적 추가.

---

## 구조

- **Good Patterns** — 재사용·권장 패턴
- **Anti-Patterns** — 자주 발견되는 문제 패턴 (AP-XX 번호 부여)

---

## Good Patterns

### GP-01: 역할-범위-제약 3단 구조
```
# 역할
너는 [명확한 역할]이다.

# 범위
[무엇을 한다 / 무엇은 하지 않는다]

# 핵심 제약
[절대 금지 사항]
```
**효과**: 클로드가 첫 문단만 읽어도 작동 방향을 잡을 수 있음.

---

### GP-02: 스킬 위임 구조
```
# [기능명]
→ [skill-name] 스킬 참조
```
**효과**: 시스템 프롬프트를 짧게 유지하면서 기능은 스킬로 확장.
시스템 프롬프트 자체가 AP-03(희석 효과)에 걸리지 않도록 방지.

---

### GP-03: 전문가 에스컬레이션 블록
```markdown
## 전문가 에스컬레이션

이 프로젝트 혼자 해결하지 않는 것들이 있다.

**클마(CGM)에게 가져갈 것:**
- 이 프로젝트의 지침(시스템 프롬프트) 수정·고도화
- KB 파일 구성 변경
- 새 클로드 프로젝트 설계 문의
- 클로드 활용 방식 전반에 대한 자문

→ claude.ai에서 CGM 프로젝트를 열고 질문한다.

**OG(OpenClaw Grand Master)에게 가져갈 것:**  ← OpenClaw 프로젝트가 있을 때만 포함
- OpenClaw 관련 모든 기술 자문
- OpenClaw 아키텍처·설정·버그·고도화

→ OG 프로젝트를 열고 질문한다.
```
**효과**: 각 프로젝트가 자기 범위를 벗어나는 질문을 적절한 전문가로 라우팅.
프로젝트 간 역할 경계가 명확해지고, 지침 품질이 유지됨.
**적용 위치**: 모든 클로드 프로젝트 시스템 프롬프트 하단에 기본 삽입.
**조건**: OpenClaw 무관 프로젝트면 OG 항목 생략.

## Anti-Patterns

### AP-01: 역할 정의 없는 시스템 프롬프트
```
❌ "당신은 도움이 되는 AI입니다."
✅ "너는 [구체적 역할]이다. [범위]. [제약]."
```
**영향**: 클로드가 범용 모드로 작동, 컨텍스트 활용 안 됨.

---

### AP-02: 모순된 지침
```
❌ "항상 짧게 답해라" + "모든 배경 설명을 포함해라"
✅ "기본은 짧게. 복잡한 질문은 [N]줄 이내로 설명 추가."
```
**영향**: 클로드가 어떤 지침을 따를지 임의 선택.

---

### AP-03: 희석 효과 (과도하게 긴 프롬프트)
```
❌ 3,000자 이상 단일 블록
✅ 스킬로 분리, 시스템 프롬프트는 핵심만 (1,500자 이내 권장)
```
**영향**: 뒷부분 지침 준수율 저하.

---

### AP-04: 구버전 XML 과용
```
❌ <prompt><role>...</role><instructions>...</instructions></prompt>
✅ 마크다운 헤더(#) 구조 사용
```
**영향**: 최신 클로드에서 불필요한 파싱 부하.

---

### AP-05: 스킬 시스템 미활용
```
❌ 모든 것을 시스템 프롬프트 한 파일에 몰아넣음
✅ 기능별 스킬 분리, 시스템 프롬프트에서 참조
```
**영향**: 유지보수 불가, 확장 불가.

---

### AP-06: 지식 누적 체계 부재
```
❌ 매번 처음부터 시작
✅ case-log, lessons, patterns 파일로 누적
```
**영향**: 프로젝트 품질 정체.

---

### AP-07: Prefilling 사용 (구시대적 기법)
```
❌ API 호출 마지막에 {"role": "assistant", "content": "네, 분석하면:"} 삽입
✅ 지침으로 출력 형식을 유도. 자유로운 Thinking Blocks 허용.
```
**영향**: Claude 4.6에서 Thinking Blocks 방해 → 추론 품질 저하. 지원 중단 진행 중.
**심각도**: 🟡 Major (Critical 진입 중)

---

### AP-08: 보안 설계 부재 (고객 대면 서비스)
```
❌ 시스템 프롬프트와 사용자 입력을 구분 없이 하나의 텍스트 블록으로 처리
✅ <instructions>지시사항</instructions> + <input>사용자 입력</input> XML 격리
```
**영향**: 간접 프롬프트 주입 공격에 무방비. 민감 정보 탈취, API 남용(Token Exhaustion) 위험.
**적용**: 고객 대면 서비스를 구축하는 프로젝트에만 적용. 내부 도구는 해당 없음.
**심각도**: 🔴 Critical (고객 대면 서비스에서 발견 시)
