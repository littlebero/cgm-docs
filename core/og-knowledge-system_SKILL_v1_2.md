# OG Knowledge System

발견한 것을 잃지 않는 방법. 어떤 도메인의 프로젝트에도 동일하게 적용된다.

---

## 1. 누적 원칙

```
버그/함정 발견 시 순서:
  1. 버그 등록 파일에 추가
  2. 교훈 파일에 교훈 추가 (동시에)
  3. 핵심 레퍼런스 해당 섹션 반영
  4. 사용자에게 알림

리서치 결과 발견 시:
  1. 리서치 로그에 누적
  2. 핵심 레퍼런스에 반영
  3. 사용자에게 알림
```

파일 업데이트 후 반드시 사용자에게 알린다.
모든 파일은 md 형식으로 생성한다.

---

## 2. 리서치 로그 형식

```markdown
## [날짜] | [주제]
출처: [URL 또는 출처]
핵심 발견: ...
CGM 영향: ...
액션: ...
```

---

## 3. 교훈 기록 형식

```markdown
## [날짜]-[순번] | [교훈 제목]
상황: ...
실수/함정: ...
올바른 방법: ...
지침 반영 여부: [반영됨 / 미반영 / 검토 중]
```

날짜 형식: `YYYY-MM-DD-NNN` (당일 순번 3자리, 예: 2026-03-02-001)
같은 날 여러 교훈이 쌓일 경우 순번으로 시간순 추적 가능.

---

## 4. 버그/이슈 등록 형식

```markdown
## BUG-[번호] | [제목]
상태: 🔴 OPEN / 🟡 WORKAROUND / 🟢 FIXED
증상: ...
원인: ...
해결: ...
```

상태 범례:
```
🔴 OPEN       — 미해결, 현재 영향 있음
🟡 WORKAROUND — 해결책 있음, 공식 픽스 미완료
🟢 FIXED      — 공식 픽스 완료
```

---

## 5. 파일 관리 원칙

- 누적형 파일(로그, 레지스터)은 삭제하지 않는다
- 같은 내용을 여러 파일에 중복 기재하지 않는다 — 원본 하나, 나머지는 참조
- 업데이트 후 반드시 사용자에게 알린다
- 형식이 없으면 먼저 형식을 제안하고 확인받은 뒤 기록한다

Knowledge Base 파일 수 관리:
```
12개 이하 유지.
초과 시 파일 수 기반 RAG 조기 활성화 위험 (토큰 수와 무관).
RAG 모드에서는 파일 간 크로스 참조 품질이 저하된다.
분기별 1회 미참조 파일 정리 권장.
```

교훈 승격 원칙:
```
case-log의 교훈 필드   → 해당 케이스에만 해당하는 특수 상황 메모
lessons.md             → 2건 이상 케이스에서 반복 확인되거나 구조적으로 중요한 교훈

승격 흐름:
  case-log 교훈 → 동일 패턴이 2회 이상 반복되면 lessons.md로 승격
  승격 시 case-log 교훈 필드에 "→ lessons.md 승격 (날짜)" 메모 남김
```
