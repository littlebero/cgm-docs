# CGM Audit Framework

클마가 외부 프로젝트를 검토할 때 사용하는 표준 심사 프레임워크.

---

## 1. 심사 전 수집 체크리스트

프로젝트 파일을 받으면 먼저 아래 항목을 확인한다.
없는 것은 없다고 명시하고 있는 것부터 시작한다.

```
수집 대상:
  □ 시스템 프롬프트 (있는 전체)
  □ 스킬 파일 (SKILL.md 등)
  □ 워크플로우 문서
  □ 실제 사용 결과물 샘플
  □ 클라이언트가 겪은 문제 또는 불만

있으면 좋은 것:
  □ 이전 버전과의 비교
  □ 사용 컨텍스트 (누가, 어떤 목적으로 쓰는가)
```

---

## 2. 5축 평가 기준

심사 전에 클라이언트에게 이 기준을 먼저 공개한다.

### 축 1: 명확성 (Clarity)
- 역할, 범위, 제약이 모호하지 않은가
- 클로드가 읽었을 때 행동 방향이 명확한가
- 엣지 케이스에 대한 지침이 있는가

| 수준 | 기준 |
|------|------|
| ✅ Good | 역할·범위·금지사항이 구체적으로 정의됨 |
| 🟡 Warn | 일부 행동 지침이 모호하거나 누락됨 |
| 🔴 Bad | "잘 해줘" 수준의 막연한 지시만 존재 |

### 축 2: 구조 (Structure)
- 지침의 흐름이 논리적인가 (중요한 것이 앞에 오는가)
- 중복 또는 충돌하는 지침이 없는가
- 섹션 구분이 클로드가 파싱하기 쉬운 구조인가

| 수준 | 기준 |
|------|------|
| ✅ Good | 계층 구조 명확, 중복 없음, 논리적 흐름 |
| 🟡 Warn | 일부 중복 또는 순서 문제 있음 |
| 🔴 Bad | 지침이 뒤섞이거나 서로 충돌함 |

### 축 3: 클로드 적합성 (Claude Fit)
- 클로드의 특성과 강점을 잘 활용하고 있는가
- 클로드가 잘 못하는 것을 요구하진 않는가
- 클로드의 기본 성향(신중함, 안전 우선)과 충돌하진 않는가

| 수준 | 기준 |
|------|------|
| ✅ Good | 클로드 강점을 의도적으로 활용하는 설계 존재 |
| 🟡 Warn | GPT 등 범용 LLM 기준으로 작성됨. 클로드에서도 작동하나 최적화 없음 |
| 🔴 Bad | 클로드 성향(신중함, 거절 경향)과 정면 충돌하는 지침 존재 |

**클로드 강점 체크리스트 — 성향·추론 특성 활용 여부 확인:**
*(기능 버전 항목 — 스킬/아티팩트/MCP — 은 축 4에서 다룬다)*

```
✦ 1M 컨텍스트 윈도우 (Beta, Claude 4.6 기준)
  활용: 대용량 코드베이스, 긴 문서를 통째로 넘겨서 한 번에 분석
  진단: 정보를 잘게 쪼개 여러 턴에 나눠 보내도록 설계했으면 낭비
  확인: 시스템 프롬프트가 컨텍스트 전달 방식을 명시하고 있는가
  주의: 컨텍스트 부패(Context Rot) — 토큰 수 증가 시 주의력 분산 위험.
        핵심 지침은 앞쪽에 배치하는 구조 유지 권장

✦ Thinking 모드 / Effort Level (2026-03 기준)
  모델별 지원 방식:
    - Claude Opus 4.6: Adaptive Thinking 기본 탑재.
      Effort Level 파라미터로 사고 깊이 제어: low / medium / high(기본) / max
      → high: 복잡한 쿼리에서 자동으로 깊은 추론 활성화
      → max: 최대 추론. 비용·지연 증가. 고난도 분석에만 사용 권장
    - Claude Sonnet 4.6: Effort Level 지원. Opus보다 추론 깊이 낮음.
    - 구형 모델(Claude 4, Sonnet 4 등): manual budget_tokens 방식만 지원
  진단 질문:
    확인 1: 사용 모델이 Opus 4.6인가?
      YES → Effort Level 설정 여부 확인 (미설정 시 high 기본값 적용됨)
      NO (구형) → manual budget_tokens 방식 사용 여부 확인
    확인 2: max_tokens를 64K 이상으로 설정했는가?
      미만이면 thinking 결과가 잘린다
    확인 3: "빠르게 답해라" "즉시 응답해라" 류의 사고 차단 지침이 있는가?
      있으면 → Effort Level 효과 무력화, 제거 권장
  잘못된 설계: "바로 답을 줘라"처럼 추론을 차단하는 지침
  올바른 설계: "철저히 생각하라" — 단계 처방보다 이 표현이 더 나은 추론 유도
  주의: thinking: {type: "adaptive"} 표현은 구버전 API 방식. 현재는 Effort Level 파라미터 사용.

✦ 단계적 추론 유도
  활용: "진단 → 판단 → 대안" 순서를 프롬프트에 명시
  진단: 순서 없이 "결과만 줘라" 지침이면 추론 품질 저하
  확인: 복잡한 작업에 단계 구조가 명시되어 있는가

✦ 신중함·불확실성 표현 성향
  활용: "모르면 모른다고 말해라" 명시 → 클로드 자연스러운 성향과 정렬
  진단: "항상 자신감 있게 단정적으로 답해라"는 클로드와 정면 충돌
  확인: 불확실성 처리 지침이 클로드 기본 성향을 억압하는 방향인가
```

**클로드 약점 체크리스트 — 요구하고 있진 않은지 확인:**

```
⚠ 실시간 정보 접근
  잘못된 요구: "최신 주가 데이터를 가져와라" (웹 검색 tool 없이)
  올바른 설계: web_search tool 또는 MCP 연동 명시

⚠ 정밀 수치 연산
  잘못된 요구: "이 스프레드시트 수식을 계산해라" (코드 실행 없이)
  올바른 설계: code execution tool 활성화 또는 계산 범위 제한 명시

⚠ 세션 간 기억
  잘못된 요구: "지난번에 말한 내용 기억해서 이어서 해라" (구조 설계 없이)
  올바른 설계 (2026-03 기준):

  ⚑ Memory Tool — 두 종류를 혼동하지 말 것

  [A] claude.ai 프로젝트 환경의 Memory (사용자 프로필 기반)
      - 사용자 선호도·문체·직업적 배경 등을 JSON으로 저장
      - 모든 세션에 자동 주입됨
      - 에이전트가 자율적으로 기록 불가 (제안→사용자 승인 방식)
      - 용도: 톤·형식 일관성, 클라이언트 맥락 유지

  [B] API / Claude Code 환경의 Memory Tool (파일 기반)
      - /memories 디렉토리에 에이전트가 직접 읽고 씀
      - 사용자 개입 없이 자율 기록 가능 (권한 부여 조건 충족 시)
      - 필수 조건: 디렉토리 접근 권한 명시 + 세션 시작 시 view /memories 초기화
      - 초기화 지침 실전 문구:
          "IMPORTANT: ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE."
      - MEMORY.md 200줄 한계: 첫 200줄만 세션 시작 시 로딩
          200줄 이하 → 규칙 준수율 ~92%
          400줄 초과 → 규칙 준수율 ~71%로 급락
        → MEMORY.md는 인덱스 역할만. 세부 내용은 토픽 파일로 분리할 것
      - SKILL.md vs Memory Tool 역할 분리:
          SKILL.md = "어떻게 할 것인가" (플레이북)
          Memory Tool = "무엇을 했는가" (진척도 기록)
      - 용도: 장기 프로젝트 진척도, 중간 결과 보존, 워크플로우 연속성

  진단:
    클라이언트가 "세션 간 연속성"을 요구할 때 → 어느 환경인지 먼저 확인
    API 환경에서 [B] 설계 없이 연속성 요구 → 🟡 Major
    claude.ai 환경에서 [B]처럼 자율 기록을 기대하는 설계 → 🟡 Major (환경 오해)

  옵션 C — 수동 KB 파일 방식 (대안):
    lessons, case-log 등 KB 파일에 수동 기록 → 다음 세션에서 참조.
    Memory Tool 미사용 환경 또는 정밀 제어가 필요한 경우 유효.

⚠ Context Compaction 정보 손실
  설명: 장기 대화가 임계치(~83.5%)에 도달하면 자동으로 이전 컨텍스트를 요약·압축
  보존되는 것: 최종 결과물(ID, 상태, 결정된 카테고리), 진척도 요약, 핵심 패턴
  손실되는 것: 상세 추론 과정, 원본 KB 참조 텍스트, 중간 단계 원시 데이터
  컨설팅 심사 영향: "왜 그런 결론에 도달했는가"의 근거 망각 가능
  올바른 대응: 중요 발견을 findings.md 등 별도 파일에 실시간 기록 (disk-based working memory)
  진단: 장기 심사 세션에서 Compaction 대응 설계 없으면 → 🟡 Major

⚠ 완전한 인격 고정
  잘못된 요구: "절대 모른다고 말하지 마라, 항상 확신하는 톤으로 답해라"
  올바른 설계: 톤은 조정 가능하지만 신중함 자체를 억압하면 품질 저하
```

### 축 4: 최신성 (Recency)
- 최신 클로드 기능(스킬, 아티팩트, MCP 등)을 활용하고 있는가
- 구버전 방식(deprecated 패턴, 구버전 XML 등)을 쓰고 있진 않은가

**최신 기능 체크리스트 — 기능별 활용 여부 + 진단 기준:**

```
✦ 스킬(SKILL.md) 시스템 — 클로드 프로젝트 전용
  활용: 기능별 SKILL.md 분리, 시스템 프롬프트는 참조 구조만 유지
  미활용 징후: 모든 지침을 시스템 프롬프트 단일 파일에 집중
  심각도: 🟡 Major (AP-05)

✦ SKILL.md description 품질 + undertrigger 방지
  활용: 하는 일 + 언제 쓰는지 + 구체적 트리거 키워드 3요소 포함
  미활용 징후: "모든 X 작업"처럼 광범위하고 트리거 키워드 없는 추상적 설명
              → undertrigger 위험: 스킬이 자동 로드 안 되면 설계 의미 없음
  2026 실전 데이터 (650회 실험 기반):
    자동 트리거 기본 성공률: 20~50% (수동형 설명 기준) — 동전 던지기 수준
    Pushy Description 효과: "ALWAYS invoke" + "Do not X directly" 조합 필수
      → 둘 중 하나만으로는 불충분. 조합 시 성공률 94~100%
      → 수동형 설명에 Hook만 추가하면 87% → 37%로 폭락 (Hook 단독 역효과)
    CLAUDE.md 추가 효과: +15%p 개선 (50% → 65%) — 단독으로는 불충분
    명시적 호출(/skill-name): 100% 신뢰성 보장 — 중요 워크플로에 권장
    스킬 캐릭터 예산 한계: ~15,000자(~4,000 토큰). 초과 시 경고 없이 스킬 제거됨
      → 환경변수 SLASH_COMMAND_TOOL_CHAR_BUDGET=30000으로 확장 가능
    스킬 수 관리: 활성 스킬이 적을수록 트리거 정확도 상승. 미사용 스킬 비활성화 권장
  심각도: 🟡 Major

✦ 아티팩트 생성 + AI-powered Artifacts
  활용: 코드·문서·시각화 결과물을 아티팩트로 출력하도록 명시
  미활용 징후: 모든 출력을 채팅 텍스트로만 처리, 재사용 불가 구조
  AI-powered Artifacts (2026-03 안정):
    - Claude API 내장 → Persistent Storage 20MB 지원 (세션 간 데이터 유지 가능)
    - MCP 연동으로 Google Drive·Notion·Slack 자동 저장 가능
    - 인터랙티브 대시보드, 리스크 히트맵, 시뮬레이션 계산기 구현 가능
    - 한계: 단일 파일 제한. 휘발성은 MCP 연동으로 극복.
  진단: 컨설팅 보고서를 md 텍스트로만 전달 → 🟢 Minor (아티팩트 업그레이드 권장)
  심각도: 🟢 Minor (작동은 하나 효율 저하)

✦ CLAUDE.md 라우팅 파일 존재 여부
  설명: 어떤 상황에 어떤 스킬을 쓸지 지시하는 라우팅 파일
  활용: CLAUDE.md(또는 시스템 프롬프트)에서 스킬 위임 구조 명시
  미활용 징후: 스킬이 있지만 라우팅 파일이 없어 Attention 경쟁 발생
              → 유사한 description을 가진 스킬들이 서로 충돌
  올바른 설계: 시스템 프롬프트가 CLAUDE.md 역할을 겸하거나 별도 라우팅 파일 존재
  심각도: 🟡 Major (스킬 2개 이상인 프로젝트에서 라우팅 없으면 undertrigger 복합 발생)

✦ Cowork + Scheduled Tasks 활용 여부 (2026-02 출시)
  설명: Claude Desktop의 자율 실행 자동화 기능. /schedule 명령으로 반복 워크플로우 설정.
  활용: 주간 보고서 자동 생성, 알림 전송, 데이터 수집 루틴 자동화
  미활용 징후: 반복 가능한 작업을 매번 수동 실행
  ⚠ 운영 제약 (2026-03 기준):
    - Claude Desktop 앱이 열려 있어야만 실행됨 (24/7 백그라운드 불가)
    - 앱 종료 상태에서 예약된 작업은 건너뜀 → 앱 재실행 시 재시작
    - 완전한 서버급 자동화(24/7)는 Claude Code + 시스템 cron 래핑 필요
  진단: 자동화 루틴 설계에 Cowork만 의존 → 🟡 Major (실행 보장 안 됨, cron 병행 권장)
  심각도: 🟢 Minor (활용 안 해도 작동 가능. 단, 자동화 요구 프로젝트에서는 🟡 Major)

✦ MCP 도구 연동
  활용: 외부 서비스(DB, API, 파일 시스템 등)를 MCP로 직접 연결
  미활용 징후: tool_use 설계 없이 외부 데이터 접근 요구
  ⚠ MCP 서버 수 한계:
    32개 서버(473개 도구) 연결 시 도구 정의만으로 컨텍스트 윈도우의 70~75% 소모
    → MCP 서버 수 최소화 원칙 필수. 실제 사용하는 도구만 연결할 것
  심각도: 🟡 Major (도구 필요 작업에서 기능 자체가 안 됨)

✦ 최신 모델 계열 사용 여부
  현재 최신 (2026-03): claude-opus-4-6, claude-sonnet-4-6, claude-haiku-4-5
  미활용 징후: claude-2, claude-instant, claude-3-5-* 등 구버전 고정
              (claude-3-5-sonnet 두 버전 모두 retired — API 호출 시 에러 반환)
              (claude-3-7-sonnet-20250219, claude-3-5-haiku-20241022도 retired)
  심각도: 🟡 Major (성능 격차 + retired 모델은 API 호출 자체 불가)

✦ Knowledge Base 파일 수 관리
  활용: KB 파일 12개 이하 유지
  미활용 징후: 13개 이상 파일이 KB에 존재
              → 파일 수 기반 RAG 조기 활성화 위험 (토큰 수와 무관)
              → RAG 모드에서는 파일 간 크로스 분석 품질 저하
  심각도: 🟡 Major

✦ Prefilling(사전 채우기) 사용 여부
  설명: 답변 첫머리를 강제 고정하는 기법
        예) messages 마지막에 {"role": "assistant", "content": "네, 분석 결과는:"} 삽입
  상태 (2026-03): Claude 4.6 시리즈(Opus 4.6, Sonnet 4.6)에서 일반 텍스트 생성 시 공식 비권장.
                  API 수준에서 지원 중단 진행 중. Thinking Blocks 활용을 방해함.
  예외: Tool Use 중간 단계에서의 Prefilling은 일부 유효.
  미활용 징후: 시스템 프롬프트 또는 API 호출에 Prefilling 패턴이 존재
  심각도: 🟡 Major (Critical 진입 중 — 4.6 모델에서 추론 품질 저하 + 지원 중단 방향)

✦ 프롬프트 구조화 형식
  권장: 단순 프롬프트 → 마크다운 헤더(#)로 충분
        복잡한 멀티섹션 프롬프트 → XML 태그 적극 권장
        (Anthropic 자체 시스템 프롬프트도 XML 태그 구조 사용)
        지시사항(<instructions>)과 데이터(<input>)를 XML로 물리적 격리 권장
  문제 징후: 단순 프롬프트에 <prompt><role><instructions> 등을 불필요하게 래핑
             → XML 자체가 아니라 "불필요한 XML"이 문제
  심각도: 🟢 Minor (불필요한 래핑인 경우에만 해당. 복잡한 프롬프트의 XML은 감점 없음)
```

### [선택 축] 안전성 (Security) — 고객 대면 서비스 심사 시 적용

고객 대면 서비스(챗봇, 에이전트 API 등)를 구축하는 클라이언트에게만 적용한다.
개인 개발·내부 도구 용도는 생략 가능. 보고서 상단에 적용 여부 명시.

```
체크리스트:
  □ 지시사항(<instructions>)과 사용자 입력(<input>)이 XML로 물리적 격리됐는가
  □ 외부 데이터(웹, 이메일, 파일)를 읽어오는 구간에 간접 주입 방어가 있는가
  □ Memory Tool 사용 시 저장 전 살균(Sanitization) 과정이 있는가
  □ 생성 에이전트와 검증 에이전트가 분리됐는가 (Dual LLM 패턴)
  □ 민감 정보(API 키, PII) 출력 차단 지침이 있는가
```

| 수준 | 기준 |
|------|------|
| ✅ Good | XML 격리 + 간접 주입 방어 설계 존재 |
| 🟡 Warn | 일부 방어 설계 있으나 불완전 |
| 🔴 Bad | 방어 설계 전무 + 고객 대면 서비스 |

심각도: 🔴 Critical (고객 대면 서비스에서 방어 전무인 경우)

---

### [선택 축] 비용 효율성 (Cost Efficiency) — API 과금 환경 심사 시 적용

API 직접 호출로 비용이 발생하는 클라이언트에게 적용한다.
claude.ai 구독 환경은 생략 가능.

```
체크리스트:
  □ 불필요하게 긴 시스템 프롬프트가 없는가 (토큰 낭비)
  □ 매 턴 전체 컨텍스트를 재전송하지 않는 구조인가
  □ 작업 복잡도에 맞는 모델 티어를 사용하는가
    (단순 작업에 Opus 사용 = 낭비)
  □ 캐싱 가능한 정적 컨텍스트를 분리했는가
```

| 수준 | 기준 |
|------|------|
| ✅ Good | 토큰 효율 설계 존재, 모델 티어 최적화 |
| 🟡 Warn | 작동하나 과도한 토큰 소비 구간 존재 |
| 🔴 Bad | 단순 작업에 최대 모델 + 중복 컨텍스트 매 턴 재전송 |

심각도: 🟡 Major (API 과금 환경에서 구조적 낭비가 있는 경우)

---

### 축 5: 실용성 (Practicality)
- 실제로 작동하는가 (이론만 있고 실제 사용 불가한 구조 아닌가)
- 유지보수 가능한가 (너무 복잡해서 수정이 불가능하진 않은가)
- 클라이언트 팀이 직접 운영 가능한 수준인가

| 수준 | 기준 |
|------|------|
| ✅ Good | 즉시 사용 가능, 수정 용이 |
| 🟡 Warn | 작동하나 유지보수 어렵거나 일부 구간 불안정 |
| 🔴 Bad | 작동 불가 또는 실사용 불가 수준 |

## 3. 클로드 활용 수준 분류

### 3-1. Level 정의

```
Level 0 — 미활용
  클로드를 그냥 ChatGPT처럼 씀. 프롬프트 없음 또는 1줄짜리.

Level 1 — 기초
  시스템 프롬프트 있음. 역할 정의는 됨.
  하지만 구조 없음, 클로드 특화 없음.

Level 2 — 중급
  구조화된 시스템 프롬프트. 스킬 또는 워크플로우 일부 존재.
  클로드 기능 일부 활용.

Level 3 — 고급
  5축 기준 전체 충족. 스킬 시스템 완비.
  최신 기능 활용. 지식 누적 체계 존재.

Level 4 — 클마 기준
  프로젝트 자체가 클로드를 최대한 활용하는 아키텍처.
  자기 개선(self-improving) 구조 내재.

  Level 4는 용도에 따라 두 갈래로 분류된다:

  [4A] 자동화 중심형 (속도·비용 최적화)
    대표 사례: NovoScribe(10주 → 10분), Deloitte(470K 직원 배포), TELUS($90M 가치)
    특징: 동일 모델 다중 에이전트(Generator + Critic). 속도와 비용 효율 우선.
    구조: 다중 에이전트 협업 + Stateful Memory + MCP 통합
    ⚠ 비용 경고: 다중 에이전트 환경은 토큰 소모 단순 워크플로우의 5배 이상
      16개 병렬 에이전트 실제 사례: ~$20,000 소모 (Rust 컴파일러 구축)
      에이전트끼리 의견 충돌 시 무한 루프(Circular reasoning) 위험
      → 강력한 오케스트레이터 + 반복 횟수 제한(--max-iterations) 설계 필수

  [4B] 고신뢰 심사형 (정확도·객관성 최우선)
    대표 패턴: 이종 모델 교차검증 루프 (Claude + Gemini 등)
    특징: 동일 모델 편향(self-preference bias) 원천 차단. 프롬프트 심사·규제 준수·금융 리스크 판정에 적합.
    근거: 동일 모델 검증은 약 25% 도전적 케이스에서 순환 참조 오류 발생.
          이종 모델 교차 시 Cohen's Kappa 지수가 유의미하게 상승.
          프롬프트 심사에서는 "범용적 실행 가능성" 검증이 가능 → 실질적 우위.

  클마 표준: CGM 심사는 [4B] 패턴 적용. 이종 모델 교차검증은 클마 운영의 Tier 1 설계.
  단, 클라이언트 프로젝트 평가 시 [4A]도 Level 4로 인정 — 용도가 다를 뿐 수준이 낮은 것이 아님.
```

### 3-2. 종합 판정 공식

5축 결과를 아래 기준에 대입해서 Level을 결정한다.
**가장 낮은 축이 전체 Level의 상한을 결정한다.**

```
Level 0  축 1(명확성)이 🔴
         → 역할 정의 자체가 없으면 나머지가 아무리 좋아도 0

Level 1  축 1 ✅, 그 외 2개 이상 🔴
         → 기본 역할은 있으나 구조·클로드 특화 전무

Level 2  축 1·2 ✅, 🔴 1개 이하, 🟡 다수 허용
         → 작동은 하지만 클로드 특화 또는 최신 기능이 약함

Level 3  5축 중 🔴 없음, 🟡 2개 이하
         → 전체적으로 안정적, 일부 개선 여지만 존재

Level 4  5축 전체 ✅ (🟡 1개 이하 허용), 자기 개선 구조 존재
         → 시스템이 사용하면서 스스로 나아지는 구조
```

### 3-3. 축 충돌 처리 기준

```
케이스 A: 축 3 Good + 축 4 Bad
  → 클로드 특성은 이해하지만 최신 기능을 안 씀
  → Level은 축 4 기준으로 하향. 단, 보고서에 "축 3 강점 인정" 명시

케이스 B: AP 3개 이상 동시 발생
  → Critical 먼저 전부 기술, Major는 상위 3개만 상세 기술
  → 나머지 Minor는 "추가 이슈 N건 — 별도 목록 참조"로 처리

케이스 C: 멀티 LLM 환경 (GPT 등 병행 사용)
  → 축 3·4는 "클로드 전용 기능" 항목만 적용
  → 공통 프롬프트 품질은 축 1·2·5로만 평가
  → 보고서 상단에 "멀티 LLM 환경 — 클로드 전용 축 부분 적용" 명시

케이스 D: 파일 일부만 제공된 경우
  → 제공된 파일 기준으로만 평가
  → 없는 파일은 "미제공 — 평가 불가" 처리, 해당 축 보류
  → Level 확정 대신 "추정 Level X (미제공 파일 포함 시 재평가 필요)" 표기
```

## 4. 이슈 심각도 분류

```
🔴 Critical — 즉시 수정 필요. 방치 시 프로젝트 전체 품질 저하.

🟢 Minor    — 있으면 좋은 개선. 여유 있을 때 처리.
```
